## Prequel

why was flutter even a thing?

this question struck me because <ExtLink href="https://reactnative.dev/">react native</ExtLink> hit its stable release almost three years before <ExtLink href="https://flutter.dev/">flutter</ExtLink> got its stable version. and yet, how did flutter compete in this space? what was with flutter that was better than react native? this was a genuine question i had to work on.

at many places, you will read that flutter is more performant than react native, alongside other blatant excuses. but is that even the real thing? 

i went deep into what problem flutter was actually trying to solve and why was it even a thing.

## Chapter 1 - The Pain

up until 2014, if you wanted to ship a high quality mobile app for android and ios, you had to write for two completely different platforms separately. you wrote java for android. then you started over from scratch and wrote objective-c for ios. it meant **maintaining two codebases, running two separate teams**, and dealing with *duplicate bugs that somehow behaved differently on each platform*.

obviously, developers wanted a shortcut. web development was huge at the time, so the obvious idea was to take standard html, css, and js, wrap it in a native shell, and ship it to both app stores.

that gave us tools like <ExtLink href="https://cordova.apache.org/">cordova</ExtLink> and <ExtLink href="https://ionicframework.com/">ionic</ExtLink>. the pitch was simple. take your web app, drop it inside a <ExtLink href="https://developer.mozilla.org/en-US/docs/Glossary/WebView">webview</ExtLink>, and you are done. one codebase.

but a <ExtLink href="https://developer.mozilla.org/en-US/docs/Glossary/WebView">webview</ExtLink> is <span className="underline decoration-dotted decoration-foreground/30 underline-offset-4">literally just a browser tab pretending to be an app</span>. scrolling always felt a bit off. *animations were clunky. gestures had weird delays.* users could feel it was **not a real native app**, even if they did not know the technical reasons why. plus, debugging rendering issues inside a wrapped browser engine was a nightmare for the developers.

so the problem was not really solved. it was just hidden until you actually tried to use the app.

## Chapter 2 - The Rise of React Native

then <ExtLink href="https://meta.com/">facebook</ExtLink> released <ExtLink href="https://reactnative.dev/">react native</ExtLink> in 2015. for the first time, cross platform development actually made sense.

instead of rendering web elements inside a browser dom, react native **mapped your javascript code directly to real native views**. a `<text>` tag became a native text renderer. a `<button>` became a real os button. it *looked and felt native because it literally was using native components under the hood*.

if you already knew react for the web, the transition was incredibly smooth. same jsx, same hooks, same component structure, just targeting native ui. it really felt like the ultimate solution to the mobile development problem.

## Chapter 3 - The Fall of React Native (not really)

but there was a **massive bottleneck in the architecture**. your javascript code ran on one thread, and the native ui ran on another. the only way they communicated was through a bridge.

every single tap, scroll, and animation frame had to be serialized into json, sent across this bridge, and deserialized on the other side. every single time.

for simple screens, it was totally fine. but the second you introduced complex gestures or heavy lists, performance tanked. frames dropped hard. the bridge was a **fundamental limit in the design**.

app sizes also got massive because all native modules were bundled by default. and whenever you needed a feature the framework missed, you had to dive into android studio or xcode, write custom native code, and wire it through the bridge manually. 

then there was the platform drift issue. you would build a screen that looked perfect on ios, but android would render the spacing differently or apply unexpected os level styles. "write once, run anywhere" often turned into *writing once and debugging everywhere*.

## Chapter 4 - The Rise of Flutter

this is where google entered the space.

most people think flutter started in 2018 when it went stable, but the real beginning was back in 2014 under a project called sky. a small team at google decided to try a completely different approach to the problem.

<CommitCard hash="00882d626a478a3ce391b736234a768b762c853a" message="open the sky" url="https://github.com/flutter/flutter/commit/00882d626a478a3ce391b736234a768b762c853a" context="adam barth's first commit back in 2014. back then it was just an internal project named sky (engine)." />

flutter **completely skipped the os widgets**. it used its own **rendering engine**, <ExtLink href="https://skia.org/">skia</ExtLink> originally, now <ExtLink href="https://docs.flutter.dev/perf/impeller">impeller</ExtLink>, and painted every single pixel directly onto a blank canvas. *it did not ask android or ios for permission to draw a button. it just drew the button itself.*

this is why we call flutter not just a framework but also a engine. it is basically a <span className="underline decoration-dotted decoration-foreground/30 underline-offset-4">game engine built for apps</span>.

because there is no bridge and no cross thread serialization, flutter hits a smooth 60fps effortlessly. animations are smooth and gestures feel instant.

this is what people really mean when they talk about performance. the goal was not just to be cross platform. the goal was to own the entire pipeline. the framework controls the language, the toolchain, and every single pixel on the screen.

## Chapter 5 - The Re-rise of React Native

a lot of developers still judge react native based on its 2018 bridge architecture. but that version is completely outdated now.

meta completely rebuilt react native from scratch. by 2024, the new architecture became the standard for new projects.

**the bridge is gone**. it was replaced by <ExtLink href="https://reactnative.dev/architecture/overview">jsi (javascript interface)</ExtLink>, which allows your js code to hold direct references to native objects in the same memory space. no more json serialization.

they also introduced <ExtLink href="https://reactnative.dev/architecture/overview">fabric</ExtLink>, a new renderer that sits closer to the metal and handles synchronous layouts without choking on heavy ui. <ExtLink href="https://reactnative.dev/architecture/overview">turbomodules</ExtLink> fixed the bloat issue by lazy loading native code only when you actually use it.

react native today is incredibly solid. the performance gap with flutter is practically invisible for most standard apps. and tools like <ExtLink href="https://docs.expo.dev">expo</ExtLink> have made the developer experience incredible. with <ExtLink href="https://docs.expo.dev/eas/">expo eas</ExtLink> handling your builds and ota updates, solo devs and small teams can ship extremely fast.

## Pre-Climax - Flutter vs React Native

let's look at the reality of choosing between them today.

**React Native, the goods**

the react ecosystem is massive. if your team already knows web development, picking up react native is straightforward. 

the biggest advantage is that it **uses real native widgets**. when apple pushes an ios update with new ui behaviors, react native apps inherit those changes automatically. you do not need to rewrite any code.

this is the reason why react native easily adopted to liquid glass coz it was really using native, while flutter couldnt. it needed a new write or a override with the existing cupertino app system.

<TweetEmbed id="1932194950403232110" />

but that native coupling is also a huge headache on android. android has dozens of custom oem skins like samsung's one ui or xiaomi's miui. react native inherits all those random styling overrides from the os, which means you sometimes get weird ui bugs that you have no direct control over.

**Flutter, the goods**

flutter took the exact opposite route. it never touches native widgets. it paints everything itself.

with the new impeller engine, performance is incredibly predictable. flutter also compiles your dart code straight into arm machine code. there is no interpreter in the middle. this is why flutter apps run smoothly even on *cheap, low ram android phones*.

because flutter controls the canvas, you get <span className="underline decoration-dotted decoration-foreground/30 underline-offset-4">absolute ui consistency</span>. your app will look exactly the same on a flagship pixel and a budget phone. no os overrides. no unexpected styling changes.

this reliability is exactly why heavy fintech apps usually pick flutter. when you have complex dashboards, charts, and intense state management requirements, flutter's rendering holds up under pressure. react native requires a lot more manual optimization to handle that level of interaction.

<ExtLink href="https://dart.dev/">dart</ExtLink> is also a great language. with sound null safety and clean syntax, any developer coming from java or kotlin will feel at home within a few days.

## Post-Climax - So Which One?

both frameworks are excellent pieces of engineering. the choice comes down to your project needs.

**Pick React Native if:** your team already knows react. you are building an ios heavy app where matching apple's native feel is crucial. you want to leverage expo for rapid deployment and ota updates. 

**Pick Flutter if:** you are building a complex, heavy app that needs to scale. you have highly custom ui designs and need absolute pixel perfect consistency across every single device. you are targeting multiple platforms beyond just mobile. or if you have a major android user base.

the biggest differentiator is that flutter owns the entire rendering pipeline from top to bottom, you are not waiting on the os to draw your ui for you.
